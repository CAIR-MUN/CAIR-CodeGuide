Consistent code style and formatting improve readability,
maintainability, and collaboration across the team. This chapter
outlines general principles and language-specific standards to follow.

## 3.1 Language-Specific Style Guides

While each programming language has its own style guidelines, the goal
is to adopt a unified formatting approach across all projects. Below are
common rules and examples.

### Python

• Use 4 spaces for indentation.

• PEP 8 is the official style guide (http://pep8.org).

• Example:

```python
def fetch_data():
    if data:
        process(data)
```

### Java

• Use 4 spaces indentation.

• Class names in PascalCase; variables and methods in camelCase.

• Example:

```java
public class DataProcessor {
    public void processData() {
        // implementation
    }
}
```

### C++

• Use 4 spaces indentation.

• Class names in PascalCase; functions and variables in camelCase.

• Example:

```cpp
class DataManager {
public:
    void loadData();
};

void DataManager::loadData() {
    // implementation
}
```

### JavaScript

• Use 2 spaces for indentation (common practice).

• Functions in camelCase; constants in UPPER_SNAKE_CASE.

• Example:

```javascript
function fetchData() {
  if (isReady) {
    process(data);
  }
}
```

## 3.2 Indentation and Whitespace Rules

Proper indentation visually separates code blocks, enhancing
readability.

• Indentation: 4 spaces for Python, Java, C++, 2 spaces for JavaScript.

• Tabs vs spaces: Use spaces; avoid tabs to ensure consistent
formatting.

• Whitespace:

• Insert spaces after commas, colons, and semicolons.

• No trailing spaces at the end of lines.

• Use blank lines to separate logical sections or functions.

## 3.3 Bracing Style

Braces define code blocks. Consistency is key.

| Language | Style | Example |
|---|---|---|
| Java & C++ | K&R style (opening brace on same line) | if (condition) { code } |
| JavaScript | Same as above or Allman style (brace on new line) | if (condition)\n{ // code\n} |

## 3.4 Line Length and Wrapping Rules

• Limit lines to 80-120 characters for clarity.

• Break long statements at logical points (after operators, commas).

• Use indentation to align wrapped lines.

## 3.5 Commenting and Inline Documentation Format

Clear comments help others understand the code.

• Use `//` or `#` for inline comments; block comments (`/* */`) only
when necessary.

• Comment why, not just what.

• Document public methods/functions with explanations of input
parameters and return values.

### Example (Python)

```python
# Fetches user data from the database

def fetch_user(user_id):
    """Retrieve user info based on user_id."""
    pass
```

## 3.6 Naming Conventions Overview

Consistent naming conventions improve code clarity.

| Language | Variables & Functions | Classes & Types | Constants |
|---|---|---|---|
| Python | snake_case | PascalCase | UPPER_SNAKE_CASE |
| Java | camelCase | PascalCase | UPPER_SNAKE_CASE |
| C++ | camelCase | PascalCase | UPPER_SNAKE_CASE |
| JavaScript | camelCase | PascalCase/Object | UPPER_SNAKE_CASE |

Examples

• Variable: `user_count`, `is_valid` (Python), `userCount`, `isValid` (Java, JS)

• Class: `DataProcessor`, `UserAccount` (all languages)

• Constant: `MAX_USERS`, `DEFAULT_TIMEOUT`
